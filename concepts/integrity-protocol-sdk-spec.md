**1.2. `intended_state` Standardization and Hashing**

*   **Schema Enforcement:** For each `action_type`, a strict JSON schema will be enforced for the `intended_state` dictionary. This addresses the "Incomplete/Misleading State" vulnerability by ensuring all critical parameters are present.
*   **Deterministic Serialization:** The `intended_state` dictionary MUST be deterministically serialized (e.g., canonical JSON) before hashing to ensure consistent `intended_state_hash` values. This prevents minor variations from invalidating commitments.
*   **Cryptographic Hashing:** A cryptographically secure hash function (e.g., SHA-256) will be used to generate `intended_state_hash`.

**1.3. C2PA-Style Provenance Signature**

*   The `provenance_signature` will be generated by the Xibalba MCP server, using a private key unique to the agent or the system.
*   The signed payload will include: `id`, `timestamp`, `agent_id`, `action_type`, `intended_state_hash`, `opa_policy_id`, and a hash of the `opa_evaluation_result`.
*   This signature provides an immutable record of the agent's intent at the time of commitment, addressing "Pre-Commitment Tampering."

#### 2. OPA Integration

**2.1. OPA Request and Response Data Model**

*   The `commit_action_intent` function will send the serialized commitment payload to the OPA service.
*   OPA will evaluate this payload against the specified `opa_policy_id`.
*   The `opa_evaluation_result` (a detailed JSON object) will be returned by OPA and embedded directly into the `BCCCommitment` object. This captures the full policy decision, not just a boolean, enabling richer auditing.

**2.2. Policy Enforcement and Denial Handling**

*   If the OPA evaluation result indicates a denial, `commit_action_intent` will raise a `CommitmentError` immediately, preventing further action.
*   The SDK will include functionality for retrieving OPA policies (read-only) for developer inspection and audit.

#### 3. Runtime Validation (Pre-Execution Check)

**3.2. Real-time Context Capture and Comparison**

*   The `actual_execution_context` MUST capture all relevant parameters and environmental variables that define the actual action being taken. This directly addresses the "Timing and Post-Commitment Intent Drift" vulnerability.
*   A lightweight, deterministic comparison algorithm will be used to check for deviations between `actual_execution_context` and the `commitment.intended_state`.
*   Any significant deviation will trigger a `RuntimeValidationError` and halt execution, preventing unauthorized actions post-commitment. The definition of "significant" will be configurable but strict by default.

**3.3. Commitment Expiration:**

*   `BCCCommitment` objects will have a short TTL (Time-To-Live) enforced by the MCP server. If `validate_and_execute` is called with an expired commitment, it will raise a `RuntimeValidationError`. This mitigates race conditions and timing attacks.

#### 5. Developer-Facing Abstractions

**5.1. Xibalba API Gateway Pattern**

*   A dedicated API Gateway service will intercept all LLM API calls (e.g., `api.xibalba.solutions/v1/chat/completions`).
*   The gateway will transparently inject commitment generation and OPA evaluation into the request flow *before* forwarding to the upstream LLM provider.
*   Developers will only need to change their `BASE_URL` and potentially add an `X-Xibalba-Action-Type` header to categorize the action for policy enforcement.
*   The gateway will return the `BCCCommitment` as part of the LLM response headers or a dedicated field, allowing developers to retrieve it for `validate_and_execute`.

**5.2. SDK Decorator/Middleware Pattern (Python Example)**

```python
from xibalba_integrity_sdk import commit_action_intent, validate_and_execute, BCCCommitment

def bcc_enforced(action_type: str, opa_policy_id: str):
    def decorator(func):
        def wrapper(*args, **kwargs):
            # 1. Capture intended state based on function args/kwargs
            intended_state = {
                "function_name": func.__name__,
                "args": args,
                "kwargs": kwargs,
                # ... other context capture logic
            }

            # 2. Commit the action intent
            commitment: BCCCommitment = commit_action_intent(
                agent_id="<dynamic_agent_id>", # Will need to be resolved at runtime
                action_type=action_type,
                intended_state=intended_state,
                opa_policy_id=opa_policy_id,
            )

            # 3. Execute with validation
            #    The action_function will be the original decorated function
            #    actual_execution_context will be captured just before calling func(*args, **kwargs)
            result = validate_and_execute(
                commitment=commitment,
                actual_execution_context={
                    "function_name": func.__name__,
                    "args": args,
                    "kwargs": kwargs,
                    # ... real-time context capture just before execution
                },
                action_function=lambda: func(*args, **kwargs) # Wrap original func for lazy execution
            )
            return result
        return wrapper
    return decorator

# Example Usage:

```

*   The `bcc_enforced` decorator will automatically wrap functions, handling `commit_action_intent` and `validate_and_execute` transparently.
*   Developers specify the `action_type` and `opa_policy_id` as decorator arguments.
*   The SDK will provide hooks or mechanisms for automatically capturing the `intended_state` and `actual_execution_context` from function arguments, class attributes, or environmental variables.
